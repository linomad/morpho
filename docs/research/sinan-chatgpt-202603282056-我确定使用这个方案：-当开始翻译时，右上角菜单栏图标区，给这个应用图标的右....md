## 一、这套方案为什么成立

## 二、我建议你直接采用的产品规范

### 1）状态机

只保留 4 个状态：

-   `idle`：默认，无原点

-   `loading`：右下角原点出现，做呼吸动画

-   `success`：不单独展示，任务结束直接回 `idle`

-   `error`：也不在菜单栏单独展示错误态，错误交给 toast / 菜单项文案

这里我建议你**不要**把 success / error 也放进菜单栏图标里。原因很简单：

-   你的菜单栏反馈职责是“我正在处理”

-   处理完成后应该立刻回归安静

-   错误如果也靠小点表达，语义太弱，用户不一定理解

所以这颗点只表示一件事：**正在翻译**。

* * *

### 2）出现/隐藏规则

建议规则直接这样定：

-   请求发出后，**延迟 150~250ms** 再显示原点

-   如果请求很快完成，在这个阈值内完成，则**不显示**

-   一旦显示，至少保持 **300ms**，避免刚出现就消失造成闪烁感

-   翻译结束后，原点淡出隐藏，不要突然消失

这个规则很重要，因为很多翻译请求其实很快。
如果你每次都立即亮点，菜单栏会一直“抖存在感”。

* * *

### 3）动画规范

你已经确定要“由小到大到小”的呼吸感，我建议参数这样收：

-   动画周期：**1.1s ~ 1.4s**

-   缩放范围：**0.82x ↔ 1.0x** 或最多到 **1.08x**

-   动画曲线：easeInEaseOut

-   不要带弹簧感，不要 overshoot

-   不要透明度大幅闪烁，最多只做轻微 alpha 变化

-   不要用纯“闪烁”，核心是**呼吸**，不是 blink

为什么这样定：
Apple 对减少运动和避免强刺激动画的建议很明确，尤其不鼓励反复、强烈、外围区域的运动。菜单栏又处在屏幕上边缘，太活跃会非常扰人。[Apple Developer+1](https://developer.apple.com/design/human-interface-guidelines/accessibility?utm_source=chatgpt.com)

* * *

### 4）原点视觉规范

建议你把原点做成：

-   位置：图标右下角内收，不要贴边

-   大小：直径约为图标可视区域的 **16% ~ 22%**

-   形态：纯圆

-   颜色：品牌色或系统强调色的低饱和版本

-   阴影：尽量不要阴影，避免发糊

-   亮度：深浅模式都要清楚，但不能刺眼

更关键的是：

**原点必须画在同一张固定尺寸的图像画布里，而不是运行时给 status item 再加一个额外子视图去撑布局。**

这点是为了保证：

-   图标占位不变

-   命中区域不变

-   相邻菜单栏图标不被挤动

-   不受 Auto Layout / intrinsicContentSize 干扰

## 三、最关键的工程原则：空间绝不能变

你已经明确要求“菜单栏图标所占空间不能发生变化”，这一点非常对。

Apple 的 `NSStatusItem` 支持固定长度；`squareLength` 表示状态项长度等于菜单栏厚度，`variableLength` 则会随内容变化。你的场景要的就是**固定长度**，不要走可变长度。[Apple Developer+2Apple Developer+2](https://developer.apple.com/documentation/AppKit/NSStatusBar?utm_source=chatgpt.com)

### 正确做法

-   创建 `NSStatusItem` 时用固定长度

-   所有状态都渲染到**同尺寸 canvas**

-   `idle` 和 `loading` 使用同样大小、同样边界的图像资源

-   只改变图像内部像素内容，不改 status item 的 `length`

### 不推荐做法

-   不要在 `button` 上临时塞一个圆点 subview

-   不要在 loading 时改 `length`

-   不要在不同状态下切换不同 intrinsic size 的图片

-   不要用文本 + 图标混排再切回纯图标

否则你会遇到：

-   图标左右跳动

-   邻近菜单项位置抖动

-   点击热区变化

-   刘海屏 / 紧凑菜单栏下更明显

## 四、我推荐的实现方式

## 方案选型：优先 AppKit，不优先 SwiftUI 直接做动画 label

如果你现在追求的是**稳定、可控、像素级一致**，我建议：

-   菜单栏项：`NSStatusBar.system.statusItem(withLength: NSStatusItem.squareLength 或固定 CGFloat)`

-   图标按钮：`statusItem.button?.image`

-   动画：自己定时更新 image，或者自定义 button layer 渲染

-   菜单内容：可以继续用 SwiftUI / NSMenu 都行

SwiftUI 的 `MenuBarExtra` 很适合快速做 menu bar app，但你这个需求是“图标内部右下角带动态装饰且占位严格固定”，AppKit 更稳、更容易精确控制。`MenuBarExtra` 当然也能做菜单栏项，但复杂动态图标细节上，我仍然建议核心状态项走 AppKit。[Apple Developer+1](https://developer.apple.com/documentation/swiftui/menubarextra?utm_source=chatgpt.com)

## 五、推荐的渲染架构

### 架构原则

不要把它理解成“菜单栏上放个会动的小圆点控件”。
要理解成：

**“我维护一张固定尺寸的菜单栏图标位图，每一帧都画出当前状态。”**

### 推荐结构

-   `StatusBarController`

    -   管理 `NSStatusItem`

    -   只接受状态更新

-   `StatusIconRenderer`

    -   输入：`isLoading`, `phase`, `appearance`

    -   输出：`NSImage`

-   `LoadingStateCoordinator`

    -   负责异步任务计数、防抖、最小展示时长

-   `AnimationDriver`

    -   用 timer / display-linked timing 推进 phase

### 动画相位

例如把一个周期抽象成 `phase: 0...1`

然后用这个 phase 去算：

-   圆点 scale

-   圆点 alpha（可选）

-   圆点直径

## 六、并发和异步下最容易踩坑的点

这是这类产品最容易做坏的地方。

你调用第三方翻译 API 是异步的，那就不能简单地：

-   开始请求：show

-   请求返回：hide

因为现实中经常会出现并发：

-   用户快速连续触发两次翻译

-   第一个请求慢，第二个请求快

-   中间有取消、重试、超时

### 正确方式：用“活动任务计数”而不是布尔值

维护一个 `activeTranslationCount`

-   新请求开始：`+1`

-   请求结束 / 失败 / 取消：`-1`

-   只要 count > 0，就保持 loading

-   count 回到 0，才触发隐藏

这样不会出现：

-   第二个请求还在跑，点却提前消失

-   某个请求失败后没清理，点常驻不灭

-   多次 show/hide 互相覆盖

### 再加两条保护

-   `count` 最低不能小于 0

-   每个 request 必须有唯一 token，避免重复 finish

## 七、你必须补上的“时序规则”

我建议你定成下面这套，不然后面体验会发散：

### 展示规则

-   `requestStart`

    -   若当前 `activeCount` 从 0 变 1，启动“延迟显示定时器”

-   延迟阈值到达且仍有活动任务

    -   进入 `loading`

-   若在阈值前全部结束

    -   不显示

### 隐藏规则

-   `activeCount` 归零时

    -   若从未显示过，直接结束

    -   若已显示，检查“最小展示时长”

    -   达到最小时长后淡出隐藏

### 建议参数

-   延迟显示：200ms

-   最小展示：350ms

-   淡出时长：120~180ms

这样你的菜单栏反馈会很稳，不会神经质。

## 八、可访问性与系统适配

这里一定要做，不然产品完成度不够。

### 1）支持 Reduce Motion

Apple 明确建议当用户开启 Reduce Motion 时，减少自动和重复动画。你可以通过系统的 reduce motion 状态来降级。[Apple Developer+2Apple Developer+2](https://developer.apple.com/design/human-interface-guidelines/accessibility?utm_source=chatgpt.com)

**降级策略建议：**

-   正常：呼吸动画

-   Reduce Motion 开启：静态小圆点，不做缩放动画

-   或者只做非常轻微 alpha 变化，但我更建议直接静态

### 2）不要只靠颜色表达

Apple 的可访问性指南强调界面不要只依赖单一方式传达信息。[Apple Developer](https://developer.apple.com/design/human-interface-guidelines/accessibility?utm_source=chatgpt.com)

所以你最好同时提供：

-   菜单栏点：视觉状态

-   菜单文案：比如菜单第一项显示 “Translating…” / “Ready”

这样即使用户看不清小点，也知道当前状态。

### 3）深色 / 浅色模式

菜单栏图标通常更适合模板图标风格；如果你用自定义绘制，也要保证在浅色和深色菜单栏上对比度足够。Apple HIG 对可见性和对比度有明确要求。[Apple Developer](https://developer.apple.com/design/human-interface-guidelines/accessibility?utm_source=chatgpt.com)

## 九、菜单栏场景下的特殊注意事项

### 1）不要假设图标永远可见

Apple 明确说菜单栏空间有限，状态项不保证始终可见，因此不能把“用户一定能看到小点”当成唯一反馈通道。[Apple Developer](https://developer.apple.com/documentation/AppKit/NSStatusBar?utm_source=chatgpt.com)

所以必须有补充反馈：

-   菜单项文案

-   可选 toast

-   可选快捷键触发后的短提示音或 HUD

### 2）用户可能隐藏菜单栏项

如果你的 app 允许隐藏菜单栏图标，或者用户从系统里移除了 menu bar extra，那么 loading 反馈就看不到了。`MenuBarExtra` 文档也提到，用户移除后 `isInserted` 会变成 `false`。[Apple Developer](https://developer.apple.com/documentation/swiftui/menubarextra?utm_source=chatgpt.com)

所以你需要定义：

-   隐藏菜单栏图标时，是否改用 dock / HUD / toast

-   或者禁止在开启“菜单栏模式”时隐藏该项

### 3）刘海屏与紧凑空间

菜单栏右侧空间在不同 Mac 上差异很大。你的图标内部细节不要太小，不然在窄空间和高分屏下会糊成一团。

## 十二、最容易出问题的 8 个坑

1.  **多个异步请求互相打架**
    用计数器，不要用单一布尔值。

2.  **请求取消后没清理状态**
    cancel、timeout、error 都要统一走 finish。

3.  **图标资源尺寸不一致**
    导致菜单栏左右跳。

4.  **动画太快太明显**
    菜单栏很敏感，轻一点才高级。

5.  **只靠颜色表达 loading**
    最好在菜单里同步文案。

6.  **未适配 Reduce Motion**
    这是完成度硬伤。[Apple Developer+2Apple Developer+2](https://developer.apple.com/design/human-interface-guidelines/accessibility?utm_source=chatgpt.com)

7.  **把原点画得太靠边**
    在某些缩放下会显得像裁切或锯齿。

8.  **把状态项当成唯一反馈渠道**
    Apple 明确不保证状态项始终可见。[Apple Developer](https://developer.apple.com/documentation/AppKit/NSStatusBar?utm_source=chatgpt.com)
